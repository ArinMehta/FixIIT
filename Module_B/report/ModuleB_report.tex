\section{Project Objective}
The objective of Assignment 2 Module B is to convert the FixIIT database design from Assignment 1 into a working full-stack database application with security, auditability, and performance-oriented improvements. The system is intended for campus maintenance management, where members can raise complaints, administrators can monitor and control operations, and the system itself can provide evidence of correctness, traceability, and optimization.

In Assignment 1, the focus was on schema design, normalization, constraints, and conceptual modelling. In Module B, the focus shifts from schema-only design to a functioning application layer built over the same schema. Therefore, the goal of this phase is not to redesign the domain, but to bring the existing schema to life with web interfaces, authenticated APIs, role-based access control, audit logging, direct-database tamper detection, and evidence-based query optimization.

The implementation was guided by the following four goals:
\begin{itemize}[leftmargin=1.5em]
    \item building a working web interface and API layer over the FixIIT schema,
    \item enforcing authentication and role-based access control locally,
    \item ensuring that every important modification is traceable and that unauthorized direct database changes can be identified,
    \item optimizing query execution using actual workload-driven indexes, measured benchmarks, and EXPLAIN-based plan analysis.
\end{itemize}

\section{System Overview}
FixIIT is a campus maintenance portal where members report infrastructure or utility issues such as electrical faults, plumbing issues, or other maintenance complaints. The final Module B implementation uses \textbf{Flask} as the backend application layer, \textbf{MySQL} as the persistent storage engine, and server-rendered HTML templates for the user-facing pages.

The Module A schema already contained the main operational entities of the maintenance domain: members, roles, tickets, assignments, comments, feedback, categories, statuses, and locations. Module B extends this by adding the necessary application support components required for a secure and verifiable system.

\subsection{Functional Scope}
The implemented application supports the following operational scope:
\begin{itemize}[leftmargin=1.5em]
    \item user login with locally validated session tokens,
    \item ticket creation and user-specific ticket listing,
    \item administrator access to global ticket management,
    \item a member profile page satisfying the ``Member Portfolio'' requirement,
    \item persistent database-backed logging of modifying operations,
    \item identification of unauthorized direct SQL modifications,
    \item performance benchmarking before and after indexing,
    \item EXPLAIN plan capture for major application queries.
\end{itemize}

\subsection{Implementation Components}
The final implementation is organized into the following layers:
\begin{itemize}[leftmargin=1.5em]
    \item \textbf{Backend routes}: implemented in \texttt{Module\_B/app/api.py}
    \item \textbf{Authentication}: implemented in \texttt{Module\_B/app/auth.py}
    \item \textbf{RBAC decorators}: implemented in \texttt{Module\_B/app/rbac.py}
    \item \textbf{Database helpers}: implemented in \texttt{Module\_B/app/database.py}
    \item \textbf{Templates}: dashboard, admin panel, and profile pages under \texttt{Module\_B/templates/}
    \item \textbf{SQL setup and index scripts}: under \texttt{Module\_B/sql/}
    \item \textbf{Benchmark and EXPLAIN artifacts}: under \texttt{Module\_B/performance/results/}
\end{itemize}

\section{Methodology and Approach}
The implementation followed an incremental, schema-preserving approach rather than a full redesign. This decision was deliberate because Assignment 1 had already produced a strong normalized schema for the maintenance domain. Therefore, Module B was designed as an application-layer realization of the existing system, with only minimal additional tables and scripts required to satisfy the new requirements.

\subsection{Reuse of Assignment 1 Schema}
The Assignment 1 schema already captured the maintenance workflow correctly, including:
\begin{itemize}[leftmargin=1.5em]
    \item member registration and identity,
    \item issue categories,
    \item ticket lifecycle through statuses,
    \item locations of complaints,
    \item assignment of technicians,
    \item comments and feedback,
    \item role associations through \texttt{member\_roles}.
\end{itemize}

Instead of modifying the domain aggressively, Module B extends it in a controlled way so that the underlying conceptual model remains consistent.

\subsection{New Additions for Module B}
To support the new assignment requirements, the following additions were made:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{Credentials}: stores username and password hash for local authentication,
    \item \texttt{member\_portfolio}: stores user profile information for the profile page,
    \item \texttt{db\_change\_audit}: stores persistent write-audit records,
    \item MySQL triggers on mutable tables to classify changes as \texttt{API} or \texttt{DIRECT\_DB},
    \item index-creation and index-removal scripts used by the benchmark workflow.
\end{itemize}

\subsection{Why the Implementation Stayed Minimal}
A key design goal was to avoid overbuilding the system. The assignment emphasizes methodology, functionality, and approach rather than unnecessary complexity. Therefore:
\begin{itemize}[leftmargin=1.5em]
    \item the existing ticketing system was preserved,
    \item the profile requirement was implemented as a domain-consistent ``My Profile'' page,
    \item audit and tamper-detection were combined into one coherent database-backed mechanism,
    \item optimization focused only on high-value queries actually used by the app.
\end{itemize}

This approach kept the implementation aligned with the assignment while maintaining consistency with the FixIIT domain.

\section{Authentication and Session Management}
\subsection{Login Flow}
The application uses a local login flow backed by the \texttt{Credentials} table. When a user submits a username and password to \texttt{POST /login}, the backend performs the following steps:
\begin{enumerate}[leftmargin=1.8em]
    \item fetch the member authentication record by joining \texttt{Credentials}, \texttt{members}, \texttt{member\_roles}, and \texttt{roles},
    \item hash the submitted password using the same SHA-256 scheme used by the stored credential row,
    \item compare the resulting hash with the stored \texttt{password\_hash},
    \item if valid, issue a JWT session token containing member identity and role metadata.
\end{enumerate}

A successful login returns:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{member\_id},
    \item \texttt{username},
    \item \texttt{name},
    \item \texttt{is\_admin},
    \item \texttt{role\_codes},
    \item \texttt{session\_token}.
\end{itemize}

This token is then used in the \texttt{Authorization: Bearer <token>} header for protected routes.

\subsection{Session Validation}
The assignment required local session validation on every API request. This was implemented using route decorators. Every protected route validates the authorization header and token before executing its underlying business logic. This ensures that requests are not trusted based on frontend behavior alone.

The final implementation was verified using route tests showing:
\begin{itemize}[leftmargin=1.5em]
    \item unauthenticated access to \texttt{/dashboard} returns 401,
    \item unauthenticated access to \texttt{/admin} returns 401,
    \item valid authenticated access to \texttt{/dashboard} succeeds,
    \item admin access to \texttt{/admin} succeeds only with an admin token.
\end{itemize}

\section{Role-Based Access Control}
\subsection{Role Model}
The system enforces two meaningful application roles for Module B evaluation:
\begin{itemize}[leftmargin=1.5em]
    \item \textbf{Admin}
    \item \textbf{Regular User}
\end{itemize}

Role information is derived from the \texttt{member\_roles} and \texttt{roles} tables and embedded in the login token payload.

\subsection{Backend Enforcement}
A major concern in systems like this is mistaking frontend hiding for actual authorization. To avoid this, role enforcement is performed on the backend using decorators:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{@login\_required} for authenticated user routes,
    \item \texttt{@admin\_required} for admin-only routes.
\end{itemize}

The final protected routes include:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{/dashboard} - login required,
    \item \texttt{/admin} - admin required,
    \item \texttt{/admin/tamper-events} - admin required,
    \item ticket update and delete routes - admin required,
    \item profile update route - login required.
\end{itemize}

\subsection{Validation of RBAC}
The RBAC mechanism was validated using direct route tests:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{/admin} without a token returned 401 Unauthorized,
    \item \texttt{/admin} with a regular-user token returned 403 Forbidden,
    \item \texttt{/admin} with an admin token returned 200 OK,
    \item \texttt{/dashboard} without a token returned 401 Unauthorized,
    \item \texttt{/dashboard} with a valid token returned 200 OK.
\end{itemize}

These tests confirm that access control is enforced by the backend and not merely suggested by the frontend UI.

\section{Member Profile Feature}
Although the FixIIT domain is a maintenance portal rather than a professional networking system, Module B explicitly required a web UI supporting a Member Portfolio. To satisfy the requirement without distorting the domain, this was implemented as a \textbf{My Profile} feature.

\subsection{Design Rationale}
The interpretation taken was that ``Member Portfolio'' in the context of FixIIT can be represented as a member-specific profile page. This keeps the implementation assignment-compliant while preserving application consistency.

\subsection{Schema Support}
A new table, \texttt{member\_portfolio}, was added with a one-to-one relation to \texttt{members}. It stores:
\begin{itemize}[leftmargin=1.5em]
    \item bio,
    \item skills,
    \item GitHub URL,
    \item LinkedIn URL,
    \item \texttt{updated\_at} timestamp.
\end{itemize}

\subsection{API Support}
The profile feature is supported by:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{GET /portfolio/me}
    \item \texttt{PUT /portfolio/me}
\end{itemize}

The implementation uses the authenticated member ID from the validated token rather than accepting an arbitrary member ID from the user. This ensures that users cannot modify another member's profile by manipulating request payloads.

\subsection{UI Integration}
The dashboard includes a \texttt{My Profile} button, which navigates to the profile page. The profile page loads member details and the editable profile fields, and allows the logged-in member to update their own information.

\section{Audit Logging}
\subsection{Need for Persistent Auditability}
The assignment required logging every modifying API action and being able to identify who changed what and when. Simple console or file logging is insufficient for this because it is not queryable at the data level and does not inherently capture before-and-after row states.

To address this, a database-backed audit table named \texttt{db\_change\_audit} was added.

\subsection{Audit Table Design}
The table stores the following fields:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{table\_name}
    \item \texttt{operation}
    \item \texttt{pk\_value}
    \item \texttt{actor\_member\_id}
    \item \texttt{endpoint}
    \item \texttt{source}
    \item \texttt{before\_json}
    \item \texttt{after\_json}
    \item \texttt{changed\_at}
\end{itemize}

This directly supports the audit requirement:
\begin{itemize}[leftmargin=1.5em]
    \item \textbf{who}: \texttt{actor\_member\_id}
    \item \textbf{what}: \texttt{before\_json} and \texttt{after\_json}
    \item \textbf{when}: \texttt{changed\_at}
\end{itemize}

\subsection{Write Coverage}
The audit mechanism was integrated with all important modifying application paths, including:
\begin{itemize}[leftmargin=1.5em]
    \item ticket creation,
    \item ticket update,
    \item ticket delete,
    \item profile update.
\end{itemize}

The backend helper sets the required audit context variables before executing each write on the same database connection.

\section{Tamper Detection via Triggers}
\subsection{Why Application Logs Alone Are Not Enough}
A log written by the application can only record actions that pass through the application. It cannot, by itself, detect a direct SQL update executed from a database client. Because Module B explicitly asks for identification of unauthorized direct database modifications, a database-level mechanism is necessary.

\subsection{Trigger Design}
MySQL AFTER triggers were added for both \texttt{tickets} and \texttt{member\_portfolio}:
\begin{itemize}[leftmargin=1.5em]
    \item AFTER INSERT,
    \item AFTER UPDATE,
    \item AFTER DELETE.
\end{itemize}

For each change, the trigger inserts a row into \texttt{db\_change\_audit}. Trigger logic inspects the MySQL session variables provided by the application:
\begin{itemize}[leftmargin=1.5em]
    \item if actor and endpoint metadata exist, the row is classified as \texttt{API},
    \item otherwise it is classified as \texttt{DIRECT\_DB}.
\end{itemize}

\subsection{Before/After Payloads}
The trigger payload behavior is:
\begin{itemize}[leftmargin=1.5em]
    \item INSERT: \texttt{before\_json = NULL}, \texttt{after\_json = NEW row}
    \item UPDATE: \texttt{before\_json = OLD row}, \texttt{after\_json = NEW row}
    \item DELETE: \texttt{before\_json = OLD row}, \texttt{after\_json = NULL}
\end{itemize}

This makes the tamper-detection mechanism simultaneously useful for auditing.

\subsection{Validation of Tamper Detection}
Two different update paths were tested:
\begin{enumerate}[leftmargin=1.8em]
    \item a manual SQL update on \texttt{tickets} generated an audit row with \texttt{source = DIRECT\_DB},
    \item an authenticated profile update through \texttt{PUT /portfolio/me} generated an audit row with \texttt{source = API}, along with actor and endpoint information.
\end{enumerate}

This confirms that the mechanism distinguishes between legitimate application activity and direct database changes.

\section{Query Optimization and Indexing}
\subsection{Optimization Philosophy}
The assignment required indexing based on actual usage patterns involving \texttt{WHERE}, \texttt{ORDER BY}, and \texttt{JOIN}. Therefore, the optimization work was driven by real application queries rather than generic database tuning.

\subsection{High-Value Query Patterns}
The main queries inspected were:
\begin{itemize}[leftmargin=1.5em]
    \item authentication lookup by username,
    \item admin-role check using member-role joins,
    \item user ticket listing with \texttt{WHERE member\_id = ? ORDER BY created\_at DESC},
    \item admin ticket listing with \texttt{ORDER BY created\_at DESC}.
\end{itemize}

Among these, the ticket listing queries were the most meaningful targets for indexing because they combine filtering and ordering in a way that is directly improved by appropriate BTREE indexes.

\subsection{Final Index Set}
The final useful indexes were:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{idx\_tickets\_member\_created\_at} on \texttt{tickets(member\_id, created\_at)}
    \item \texttt{idx\_tickets\_created\_at} on \texttt{tickets(created\_at)}
\end{itemize}

\subsection{Why These Indexes Were Chosen}
The reasoning is as follows:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{tickets(member\_id, created\_at)} supports the user-specific ticket query efficiently by combining the filter column and the sort column,
    \item \texttt{tickets(created\_at)} supports global admin listing sorted by time,
    \item redundant indexes on uniquely constrained columns were avoided because they did not add meaningful value for the observed workload.
\end{itemize}

\subsection{Redundant Index Removal}
To make the optimization defensible, clearly redundant or low-value indexes were removed or avoided in the benchmark scripts. This prevents over-indexing and aligns the report with actual optimization logic rather than cosmetic index creation.

\section{Benchmarking Methodology}
\subsection{Need for Reproducible Evidence}
Simply claiming that indexing improves performance is not sufficient. Therefore, a reproducible benchmark pipeline was built in \texttt{performance\_test.py}.

\subsection{Pipeline Steps}
The benchmark script performs the following workflow:
\begin{enumerate}[leftmargin=1.8em]
    \item run \texttt{drop\_indexes.sql} to establish the baseline state,
    \item benchmark the selected application queries,
    \item capture \texttt{EXPLAIN FORMAT=JSON} for the same queries,
    \item run \texttt{create\_indexes.sql} to enable the optimized state,
    \item re-run the same benchmarks,
    \item re-capture EXPLAIN plans,
    \item save the artifacts under \texttt{performance/results/}.
\end{enumerate}

\subsection{Generated Artifacts}
The final benchmark pipeline produces:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{timings\_before.csv}
    \item \texttt{timings\_after.csv}
    \item \texttt{explain\_before.json}
    \item \texttt{explain\_after.json}
    \item \texttt{summary.md}
\end{itemize}

This makes the optimization process reproducible and inspectable.

\subsection{Interpretation}
The benchmark shows measurable improvement across all benchmarked queries. However, not all improvements should be interpreted equally.

The most meaningful optimization evidence comes from:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{USER\_TICKETS\_QUERY}
    \item \texttt{ADMIN\_TICKETS\_QUERY}
\end{itemize}

These queries are directly aligned with the added indexes and therefore best demonstrate the value of workload-driven indexing.

The login and admin-check queries also show improvement, but the login query in particular is not the strongest demonstration of index value because MySQL simplifies parts of it aggressively during planning. Therefore, the ticket listing queries form the core optimization argument in this report.

\section{EXPLAIN Query-Plan Analysis}
\subsection{Why EXPLAIN Was Used}
The assignment explicitly required EXPLAIN or equivalent query-plan evidence. Therefore, \texttt{EXPLAIN FORMAT=JSON} was captured before and after indexing for each selected query.

\subsection{Queries Analyzed}
The EXPLAIN workflow was applied to:
\begin{itemize}[leftmargin=1.5em]
    \item LOGIN\_QUERY
    \item ADMIN\_CHECK\_QUERY
    \item USER\_TICKETS\_QUERY
    \item ADMIN\_TICKETS\_QUERY
\end{itemize}

\subsection{Interpretation of Plans}
The JSON plans confirm that the optimizer changes access behavior after indexing. The most important interpretation is not just that EXPLAIN files exist, but that the ticket queries - the most workload-relevant paths - are the clearest examples of useful indexing.

The EXPLAIN outputs were saved as:
\begin{itemize}[leftmargin=1.5em]
    \item \texttt{performance/results/explain\_before.json}
    \item \texttt{performance/results/explain\_after.json}
\end{itemize}

These files serve as evidence accompanying the benchmark summary.
